# Aerodynamics4MC Integration Notes

## Current Boundary

Aerodynamics4MC is useful to FPV Dronecraft as a world wind provider. Its public API exposes trusted gameplay wind samples with mean wind, bounded gusts, pressure anomaly/proxy, turbulence, shear, shelter, confidence, humidity, temperature, and ABL stability/mixing diagnostics. It does not replace FPV Dronecraft's multirotor model: rotor thrust, induced flow, vortex ring state, propwash, battery, motor dynamics, and airframe forces still live in `drone-sim-core`.

The first integration keeps that boundary explicit:

- `fabric-mod` samples Aerodynamics4MC only when the mod is present at runtime.
- The bridge uses `GameplayWindSample` with `GAMEPLAY_SERVER_ONLY`, so server physics ignores client-local L2 wind.
- Missing or incompatible Aerodynamics4MC falls back to FPV Dronecraft's existing Minecraft-weather wind.
- `/fpvdrone environment wind ...` remains the highest-priority override.
- The simulation core stays Minecraft-free and does not depend on Aerodynamics4MC classes.

## Upstream API Audit

The current Aerodynamics4MC Core API keeps the server gameplay entry point compatible with the bridge: `AeroMinecraftWindApi.sampleGameplay(ServerLevel, Vec3, SamplePolicy)` returns `GameplayWindSample`, and the gameplay default is still `GAMEPLAY_SERVER_ONLY`. FPV Dronecraft deliberately passes that policy explicitly so authoritative physics continues to ignore client-local/debug flow.

The bridge now tracks the full gameplay sample contract used by the simulator: mean, effective, and explicit gust vectors; pressure; turbulence; shear; shelter; updraft; source temperature and humidity; confidence; source level; authority; L1/world-delta/L2 freshness epochs; and ABL stability/mixing diagnostics. Optional reflection is kept for fields that are useful telemetry or local L2 modifiers, so an older A4MC build can degrade to unavailable values without disabling the whole wind bridge.

Aerodynamics4MC also exposes lower-level L2 request/result types with grid dimensions, cell size, time step, inlet velocity, density, viscosity, solid masks, optional flow-atlas output, and optional force/moment output through `AeroL2ForceMoment`. FPV Dronecraft now keeps a separate `Aerodynamics4McL2Bridge` capability probe for that contract. It does not change authoritative gameplay wind; it verifies whether the API-level `AeroWindApi.runL2(AeroL2Request)` path, flow-atlas sampling, solid-mask setup, and force/moment fields are available, can reflectively build a bounded `AeroL2Request` from an `L2RequestSpec` when A4MC is present, and can invoke `runL2` to extract only a compact status/flow-atlas/force-moment summary. Treat that as the next bounded CFD refinement path, not as part of the gameplay wind bridge. Future integration should run L2 force/moment requests behind availability and latency budgets, most likely for offline calibration packets or a separate server-authoritative local-flow solver, then feed only capped force, moment, pressure-center, or voxel-flow summaries into `drone-sim-core`.

## Implemented Path

`Aerodynamics4McWindBridge` reflects against `AeroMinecraftWindApi.sampleGameplay(ServerLevel, Vec3, SamplePolicy)`. It binds A4MC's explicit `gustVelocityVector()` when available and falls back to `effectiveVelocityVector() - meanVelocityVector()` for older bridge-compatible APIs. A sample with flow is preserved for telemetry, while trusted/confident/fresh samples feed `DroneEnvironment` as natural wind, including the current stage-one playable environment when Aerodynamics4MC is installed:

- mean wind becomes the base wind vector after confidence/freshness weighting against Minecraft weather or calm fallback, while explicit gust remains separated for transient source-gust adoption,
- API turbulence becomes the natural turbulence floor and barometer sensor-noise input, with the core falling back to quality-weighted raw A4MC source turbulence when direct callers provide raw telemetry before the fabric bridge has folded it into environment turbulence,
- shear and shelter add bounded dirty-air turbulence,
- trusted local L2 pressure anomaly/proxy adds a small capped turbulence boost, so block-edge or duct pressure disturbance can excite rotor response without turning coarse weather pressure into local CFD forcing,
- A4MC shear/updraft/shelter diagnostics add a bounded terrain-shear gust component to the effective air mass, with ABL stability/mixing shaping its response and decay without changing the bounded target magnitude,
- trusted A4MC updraft now also becomes its own smoothed vertical air-mass split before terrain-shear synthesis, subtracting explicit A4MC vertical gust and already adopted vertical mean wind before adoption so effective-flow `Y` is not counted twice, with ABL stability/mixing shaping adoption so mixed unstable thermals respond more strongly while stable low-mixing layers remain damped,
- gust vector, with gust-speed fallback for older bridges, adds a bounded unresolved turbulence boost plus a small coherent A4MC source-gust air-mass component whose vertical and scalar response is shaped by ABL stability/mixing; the turbulence/updraft contribution also removes explicit `gustY` and represented vertical mean wind first so vertical gust energy is not counted as both gust and updraft,
- raw API temperature and humidity are preserved for telemetry; the core applies confidence/freshness weighting to source temperature before deriving effective air temperature for density, Mach/Reynolds, icing, barometer, and thermal models, while humidity is quality-weighted inside the core before contributing to moist-air density relief,
- raw pressure anomaly is preserved for telemetry, then confidence/freshness weighted before adjusting effective air-density ratio and barometer pressure, and trusted local L2 pressure also contributes a small bounded barometer static-port bias while staying bounded for L2 proxy values,
- source level, authority, and freshness age are preserved for L0/L1/L2 regression analysis, with stale source samples faded out of base wind, atmospheric scalar, local voxel, disk-gradient, and turbulence coupling,
- raw A4MC mean/effective/gust/turbulence split is preserved alongside confidence/freshness/source-quality telemetry, while adopted base wind and core transients use the shared quality gate,
- quality-weighted A4MC humidity participates in moist-air density, bounded moist-air dynamic viscosity for rotor Reynolds/low-Re effects, thermal ventilation cooling, and low-temperature saturated-air icing, so hot humid air can change rotor authority/Reynolds behavior and slow motor/ESC/battery heat rejection while freezing fog-like humidity can add light time-accumulating rotor icing without being treated as rain film,
- trusted local L2 A4MC shelter reduces the direct precipitation exposure used by body and rotor rain-film sampling, so covered tunnel mouths and wall shadows shed some wet-prop load without changing Minecraft's rain state or water-immersion model,
- asymmetric trusted local L2 A4MC shelter/obstacle residuals and radially aligned rotor pressure-gradient wind equivalents now add a small source-availability-gated airframe pressure-center migration from adopted per-rotor signals, letting block-edge or tunnel-mouth shielding create body-scale aerodynamic moments during cruise or powered hover instead of only rotor-local obstruction,
- the core flight model now applies the same trust/confidence/freshness quality gate to A4MC source-gust, terrain-shear target/decay, and local-voxel ventilation transients, so direct `DroneEnvironment` callers cannot excite or reshape those paths with stale or untrusted samples,
- untrusted A4MC flow samples remain visible as zero-quality wind-source telemetry instead of being collapsed into `unavailable`, making client-local or otherwise non-authoritative sources auditable without letting them affect server physics,
- ABL stability and mixing modulate the core Dryden turbulence intensity/correlation time and the near-surface boundary-layer wind profile: unstable mixed layers strengthen and quicken gust response while ventilating ground/ceiling shear faster, and stable low-mixing layers damp Dryden response while leaving stronger surface shear and recirculation.

When trusted Aerodynamics4MC samples are available, `fabric-mod` samples each rotor center plus weighted disk-edge points and passes both disk-mean wind and a bounded in-plane axial wind gradient into `drone-sim-core`. The disk-edge directions are projected onto each rotor's actual thrust-axis plane before world sampling, so outward-canted or otherwise tilted rotors read local voxel flow from their real disk plane instead of a fixed body X/Z plane. The bridge explicitly requests A4MC `SamplePolicy.GAMEPLAY_SERVER_ONLY`, so server-trusted L2 can refine block-edge airflow while client-local/debug L2 remains excluded from authoritative flight physics. Missing or low-quality disk-edge samples remain in the disk quadrature as center-wind fallback, with confidence/freshness fading applied before the disk blend, making partial local voxel coverage conservative instead of amplifying one-sided samples into a full-disk gradient. The body-center source-quality check only decides whether rotor-field sampling is available; disk/local-voxel outputs keep the per-sample quality already embedded in their blends instead of multiplying by body quality again. The same disk-edge samples also calculate bounded shelter and pressure gradients: uneven A4MC shelter around a rotor disk contributes a small directional side-flow obstruction hint, while local L2 pressure differences are converted into a small equivalent disk-wind gradient that reuses the existing asymmetric rotor-loading path. Missing, stale, or coarse shelter/pressure samples fade back to the rotor-center value instead of creating a false wall direction. The separated per-rotor pressure-gradient wind-equivalent array is also treated as an adopted local-voxel signal in the core, with source freshness/trust used only to reject unavailable local flow rather than attenuating that array a second time. The rotor disk wind blend uses A4MC mean wind only, so coherent source gust remains separated for the core air-mass transient instead of being folded into both disk gradient and source-gust response. The core also uses the adopted per-rotor local-voxel shelter/residual and pressure-gradient imbalance to add a bounded airframe pressure-center offset, but the pressure-gradient portion is first projected onto each rotor arm's radial body direction; one-sided block shielding or radially aligned pressure steps can therefore create a small body-scale moment in straight cruise without being squared down by body-center confidence, while purely tangential rotor-disk pressure gradients stay rotor-local instead of moving the whole airframe pressure center. The fabric coupling adopts the quality-weighted A4MC mean wind as the base environment wind and leaves explicit gust energy for source-gust transients; it passes Minecraft-derived ambient temperature plus raw A4MC source temperature separately so the core owns the single quality-gated temperature adoption used by density, rotor aerodynamics, icing, barometer, and thermal systems. It also de-overlaps explicit vertical gust and the already adopted vertical mean wind before adding updraft-derived natural turbulence, so the environment turbulence floor mirrors the core air-mass split instead of double-counting A4MC effective-flow `Y`. For direct core callers, the same floor is derived from the greater of explicit environment turbulence and quality-weighted raw A4MC source turbulence, so raw telemetry does not need a fabric-only pre-folding step to drive Dryden-scale response or barometer sensor-noise amplitude. The core keeps its existing global Dryden and dirty-air wind model, folds the quality-weighted A4MC gust vector into a small coherent source-gust air-mass vector, shapes source-gust vertical mixing with ABL diagnostics, adopts trusted A4MC updraft as a separate smoothed vertical air-mass vector after removing explicit vertical gust already represented by the source-gust split and vertical mean flow already represented by the base wind, then applies the per-rotor wind delta to axial gust and lateral inflow calculations. ABL stability/mixing now also gates that updraft target and smoothing time: mixed unstable layers strengthen and quicken thermals or shaft-like rising flow, while stable low-mixing layers damp and slow the vertical split. The same ABL gate now shapes ground/ceiling boundary-layer wind recovery and shear-layer dirty-air/recirculation: mixed unstable layers recover horizontal wind toward freestream and shed less surface shear, while stable low-mixing layers preserve a stronger surface wind deficit and recirculation penalty. When only the gust speed scalar is available, the core preserves the earlier bounded synthetic gust direction but applies the same ABL scalar/vertical shaping. A4MC shear/updraft diagnostics still drive a small correlated terrain-shear wind component in the core air-mass model, with A4MC shelter increasing the wake-like component only when mean wind is present; ABL stability and mixing now tune that component's smoothing time so mixed unstable layers flush faster while stable low-mixing layers persist longer, and that terrain-shear path uses the same de-overlapped updraft signal. This lets cliffs, openings, and sheltered voxels produce time-varying relative-air changes even when the mean source wind is steady. A4MC humidity now feeds the core moist-air model beyond density by applying a bounded vapor-pressure-based dynamic-viscosity relief to rotor Reynolds/low-Re calculations, reducing the dry-air thermal ventilation term for motors, ESCs, and the battery pack under hot humid conditions, and contributing a bounded freezing-humidity equivalent wetness to the time-accumulating icing model when the air is both cold and near saturated. The core derives adopted temperature and humidity through the same trust/confidence/freshness gate used by A4MC transients, so stale or untrusted atmospheric scalars remain logged but cannot change density, rotor Reynolds behavior, cooling, icing, barometer pressure, or sensor turbulence noise. That freezing humidity path does not wet the prop film or trigger rain-film thrust loss, leaving water immersion and precipitation film cooling separate. The disk gradient adds first-order asymmetric disk loading: small thrust loss, extra aerodynamic load/vibration, bounded local dynamic-stall onset, and flapping tilt toward the higher axial-flow side. This keeps Minecraft-specific wind sampling outside the physics core while allowing rotor-to-rotor and disk-scale flow differences near cliffs, openings, and sheltered blocks.

Powered hover now opens that same local pressure-center path through a small rotor-wash downforce gate. A trusted asymmetric local L2 offset can therefore create a bounded body-scale moment beside blocks or tunnel mouths even when freestream airspeed is near zero, while stale, coarse, or untrusted samples still fade to no body moment.

In the advanced environment path, local obstacle scans, drone wake, water, precipitation, ground effect, ceiling effect, and rotor disk obstruction remain active on top of the sampled wind. If Aerodynamics4MC reports trusted local L2 voxel flow, FPV Dronecraft treats A4MC shelter as having already explained part of the wall/tunnel wind shadow and attenuates only the duplicated local obstacle airflow plus rotor side-flow obstruction. Rotor-center A4MC samples now refine that attenuation per rotor, so the wall-side rotor at a tunnel mouth can shed more duplicated Minecraft obstruction than the exposed rotor instead of inheriting only the body-center shelter residual. Rotor-center samples that are present but stale, coarse, or otherwise zero-quality keep the body residual fallback instead of restoring duplicate geometric obstruction abruptly. The same trusted local shelter now also attenuates precipitation exposure before the wet-prop model sees it: body-center shelter gates the body rain sample, while rotor-disk mean shelter gates each rotor's rain-film input, and no trusted local voxel sample leaves the existing Minecraft sky/rain exposure unchanged. The core still quality-gates raw body shelter, while per-rotor residual and shelter-obstruction arrays carry the confidence/freshness weighting already applied by the fabric local-voxel bridge, so half-trusted A4MC local flow fades once instead of being squared down. Rotor ground effect, ceiling effect, wet props, precipitation, and drone wake stay independent. In the stage-one playable path, the environment remains calm unless Aerodynamics4MC or an explicit `/fpvdrone environment` override provides wind or turbulence.

This gives the CSV regression suite a stable rainy block-edge/tunnel-mouth proxy before live Aerodynamics4MC worlds are available in automated tests.

The offline recorder `Summary:` line now also prints `max_pc_torque`, `max_wallfac`, and `min_a4mc_vent=rotor/pack`, letting quick CLI runs audit the same pressure-center moment peak, rotor wall-force geometry gate, and minimum A4MC ventilation-efficiency loss that the CSV columns preserve sample by sample.

`Aerodynamics4McL2Bridge` is intentionally separate from `Aerodynamics4McWindBridge`. The gameplay bridge remains locked to `GAMEPLAY_SERVER_ONLY` samples, while the L2 bridge reports whether A4MC's API-level wind-tunnel machinery is present: request builder, cell size/timestep/air/inlet controls, solid masks and initial flow fields, flow-atlas velocity/pressure sampling, and `AeroL2ForceMoment` force, moment, pressure-center, and reference-point components. It also owns the first bounded request/run path: `L2RequestSpec.forceMomentProbe(...)` creates a conservative force/moment calibration request with no full flow atlas by default, seeded uniform inlet flow, optional solid mask, centered force reference, and local checks for grid size, step count, sample stride, inlet velocity, density, viscosity, timestep, and mask length before any A4MC builder method is invoked. `run(...)` then calls A4MC only after a request has passed those checks and returns an `L2RunResult` containing invocation state, success/availability/status text, flow-atlas presence/value count, and a finite `L2ForceMomentSample` with force magnitude, moment magnitude, and pressure-center offset helpers. Missing or incompatible A4MC classes return an unavailable capability/build/run result instead of throwing, so the simulator can expose future calibration or diagnostics conditionally without adding a hard dependency.

`Aerodynamics4McL2DroneProbe` turns FPV Dronecraft's existing `DroneConfig` presets into A4MC-scale wind-tunnel requests. It derives a bounded grid and cell size from rotor span/radius, builds a binary solid mask for the static airframe body, arms, and rotor hubs, leaves the full rotor disk open instead of treating it as a solid wall, and returns both the `L2RequestSpec` and geometry metadata such as cell size, rotor span, solid-cell count, and body-coordinate mask lookup helpers. This gives APDrone, racing quad, cinewhoop, heavy-lift, and future tuned configs a shared offline airframe force/moment calibration path before any powered-rotor or porous-disk approximation is attempted.

`Aerodynamics4McL2DroneProbeAudit` packages that geometry path into a stable four-preset audit surface. The packet records racing-quad, APDrone, cinewhoop, and heavy-lift grid dimensions, mask size, solid-cell fraction, rotor span, rotor-disk resolution, force-reference center, inlet vector, force/moment request flags, body-center and rotor-hub mask coverage, and sample points on each rotor disk that must remain open. It gives future CFD coefficient work a small regression target for "did the wind tunnel geometry change?" before comparing live A4MC force and pressure-center outputs.

`Aerodynamics4McL2StaticAirframeRunMatrix` adds the first compact run-summary layer on top of that geometry. It runs the same four static-airframe force/moment probes through the reflected L2 bridge, keeps raw flow-atlas output disabled, records invocation/success/availability status, and stores only force, moment, pressure-center offset, and runtime summary text. The checked-in packet uses the test L2 API fixture to lock the reflection contract; production calls still return unavailable until the Aerodynamics4MC mod is actually loaded.

`Aerodynamics4McL2StaticAirframeCoefficientSeed` normalizes those compact run summaries into dimensionless force and moment coefficients using the probe body ellipsoid's frontal reference area, rotor span as reference length, and sea-level air density. It is a comparison seed rather than a gameplay coefficient table: test-runtime values intentionally prove the math and packet shape only, while live A4MC runs are still required before any coefficient feeds `drone-sim-core`.

`Aerodynamics4McL2StaticAirframeCoefficientAcceptanceGate` adds the safety gate between coefficient seeds and gameplay fitting. It requires all four expected preset seeds, fit-ready normalization, a live A4MC runtime label instead of the test fixture, and bounded drag, force, moment, and pressure-center ratios before any coefficient set can be considered for `DronePhysics` tuning. The checked-in packet deliberately blocks both current unavailable production runs and test-runtime seeds, while preserving one synthetic live-runtime pass scenario as the target contract.

`Aerodynamics4McL2StaticAirframeGameplayFitCandidate` records the first coefficient-to-gameplay mapping surface without changing runtime physics. It converts accepted static-airframe drag coefficients into a forward `DroneConfig.bodyDragCoefficients().z()` target, equivalent CdA, and target-to-current ratio, but only marks those rows as forward-drag candidates when the coefficient acceptance gate is open. Full sideforce, lift, moment, and pressure-center gameplay fitting remains disabled until live multi-axis A4MC L2 sweeps exist, so the current packet is an audit surface rather than an automatic `DronePhysics` retune.

`Aerodynamics4McL2StaticAirframeMultiAxisSweepPlan` fills that missing multi-axis evidence contract. For each of the four static-airframe presets it records six force/moment L2 request targets: forward drag, reverse-drag symmetry, left/right sideslip, and positive/negative angle-of-attack. Each case keeps raw flow-atlas output disabled, preserves compact force/moment/pressure-center intent, and computes the directional projected ellipsoid reference area for the inlet vector. This gives live A4MC runs a bounded matrix for sideforce, lift, moment, and pressure-center fitting before full gameplay retuning is allowed.

`Aerodynamics4McL2StaticAirframeMultiAxisRunMatrix` makes that 24-case sweep executable through the same reflected L2 bridge as the single-axis static-airframe run matrix. It rebuilds each planned request, keeps full flow-atlas output off, and stores only compact invocation, availability, force, moment, pressure-center, status, and runtime fields. The checked-in packet uses the test L2 API fixture to lock the contract shape; production remains unavailable until a live Aerodynamics4MC runtime is loaded.

`Aerodynamics4McL2StaticAirframeMultiAxisCoefficientSeed` normalizes those compact multi-axis run summaries into signed dimensionless coefficients. Each sweep uses the directional projected ellipsoid area and rotor span reference length, preserving body-axis force coefficients, signed inlet-axis force, moment coefficients, and pressure-center offset ratio. The current packet remains test-runtime-only, but it defines the coefficient surface needed before any sideforce, lift, moment, or pressure-center gameplay fit can be accepted.

`Aerodynamics4McL2StaticAirframeMultiAxisCoefficientAcceptanceGate` keeps the full static-airframe gameplay fit closed until all 24 multi-axis coefficient seeds are present, fit-ready, live-runtime sourced, bounded, and forward/reverse drag sign-consistent. The checked-in packet blocks current unavailable production runs, blocks test-runtime seeds, preserves one synthetic live-runtime pass scenario, and catches a synthetic forward/reverse sign break before any sideforce, lift, moment, or pressure-center coefficient can reach `DronePhysics`.

`Aerodynamics4McL2StaticAirframeMultiAxisGameplayFitCandidate` turns an opened multi-axis coefficient gate into an audit-only candidate mapping for the current gameplay scales. For each preset it derives X/Y/Z quadratic body-drag targets from signed axial CdA pairs, estimates sideforce and pitch-lift gains from the 12-degree sweep response, carries the L2 pressure-center offset vector through to body-frame center-of-pressure targets, records moment magnitudes, and compares all of them with the current `DroneConfig` terms. Runtime config auto-apply deliberately remains closed until a live calibration review decides how these full-airframe targets should be blended into shipped flight presets.

`Aerodynamics4McL2StaticAirframeMultiAxisConfigBlendPolicy` adds that missing review surface without changing shipped presets. It consumes only accepted multi-axis gameplay-fit candidates, caps suggested `DroneConfig.bodyDragCoefficients()` movement to 18 percent toward the A4MC target, caps suggested `centerOfPressureOffsetBodyMeters()` movement to 20 percent toward the resolved pressure-center vector, and keeps runtime config auto-apply false for every row. Static moment magnitudes are flagged as an angular-drag review requirement rather than being blended into `angularDragCoefficient`, because rotational damping still needs a dynamic rotational sweep or explicit calibration review.

`Aerodynamics4McL2StaticAirframeRotationalDampingSweepPlan` defines that dynamic angular-drag evidence contract. For each static-airframe preset it records positive and negative pitch, yaw, and roll body-rate targets, the reused static L2 geometry baseline, the representative tangential surface speed, and the current `angularDragCoefficient` torque scale. The packet deliberately marks rotational-flow API support unavailable and angular-drag calibration disallowed until A4MC can run force/moment cases with a rotational velocity field or moving-boundary equivalent, so static moment coefficients are not mistaken for body-rate damping derivatives.

`Aerodynamics4McL2StaticAirframeRotationalDampingRunMatrix` turns that rotational damping plan into a compact execution-status packet without fabricating angular-drag evidence. It carries every pitch, yaw, and roll body-rate case forward with the static baseline request geometry, but marks the current rows as `SKIPPED` with `rotational-flow-api-unavailable` until A4MC exposes a rotational velocity-field or moving-boundary request surface. If that API appears before a live executor is wired, the matrix can distinguish buildable-but-not-invoked `PENDING` rows from truly unavailable rotational evidence.

`Aerodynamics4McL2StaticAirframeRuntimeRetuneReadinessGate` combines those review layers into the final static-airframe preset-retune gate. It requires all four presets to have reviewed body-drag and pressure-center suggestions, explicit runtime preset mutation enablement, and all 24 rotational damping sweep cases to have available angular-drag evidence before any static-airframe runtime retune can open. The current audit scenarios remain blocked, while one synthetic reviewed/evidence-available scenario documents the target contract future live A4MC runs must satisfy.

`Aerodynamics4McL2RotorDiskAperture` adds the next geometry layer without changing the binary solid mask: each rotor gets a true thrust-axis disk plane, a stable radial/tangential basis, four equal-area radial rings, sixteen azimuth samples per ring, and an estimated open actuator-disk area after static airframe body/arm/hub blockage. The packet also separates rotor-axis inlet from in-plane crossflow, so forward-flight wind-tunnel cases can distinguish through-disk powered-rotor loading from edgewise rotor-disk exposure before A4MC offers or adopts a porous source-term representation.

`Aerodynamics4McL2ActuatorDiskCalibration` now turns that open rotor-disk area into a first powered-rotor source-term audit. It scans idle, hover, cruise, and max spin states for racing-quad, APDrone, cinewhoop, and heavy-lift presets, then records per-rotor thrust, total thrust-to-weight, open-area pressure jump, ideal induced velocity, far-wake velocity, momentum power, and edgewise advance ratio. This is still a calibration guardrail rather than a live A4MC source-term injection path: the intent is to validate these pressure jumps and wake speeds against future A4MC L2 force/moment and pressure-center runs before any gameplay force is driven by them.

`Aerodynamics4McL2ActuatorDiskSourceMap` converts the hover pressure-jump target into per-rotor source terms with center, thrust-axis direction, open area, force vector, and body-origin moment. For the current symmetric presets the source-map summary expects net hover force to equal vehicle weight, net moment to remain zero, and center of thrust to stay at the body origin. That gives future A4MC powered-source experiments a direct force/moment delta target instead of only aggregate disk-loading numbers.

`Aerodynamics4McL2PoweredHoverSourceMap` adds the hover-specific powered-source packet that mirrors the forward-flight cruise packet. It keeps the `hover` spin state explicit while carrying the per-rotor source terms, source center-of-thrust offset, induced/far-wake velocity, tip speed, edgewise advance ratio for the static-airframe inlet, and representative blade Reynolds number. The packet remains API-unavailable and request-target-only, but future A4MC body-force or porous-source executors can now consume hover and cruise source-map rows through the same shape instead of treating hover as a special case hidden inside the generic actuator-disk audit.

`Aerodynamics4McL2PoweredHoverWakeFootprint` derives a hover downwash footprint from that source-map target without opening runtime coupling. It records disk-plane wake area, ideal far-wake contraction to half area, equivalent per-rotor wake radii, mass flow, momentum-flux and kinetic-power closure errors, plus radius-in-cell sample targets for a future local A4MC wake probe. This keeps near-field downwash research pointed at physical validation first while still leaving a compact reference packet that a later playable branch can consult only after live powered-source and wake-probe evidence exists.

`Aerodynamics4McL2PoweredHoverSurfaceImpingement` extends that hover wake target into ground and ceiling impingement samples. For each preset it projects the contracted far-wake momentum flux onto 0.5R, 1R, 2R, and 4R surface clearances, reuses the JIRS 2024 ground/ceiling exponential curve fits from `SurfaceNearfieldCalibration`, and records the expected cushion force and reaction pressure. The packet is still audit-only: it defines what live A4MC powered-source plus local wake probes should recover near surfaces before any surface-coupled downwash result can inform gameplay feel.

`Aerodynamics4McL2PoweredHoverSurfaceProbeMap` turns those aggregate surface-impingement rows into rotor-resolved A4MC probe points. Each hover source term contributes ground and ceiling probe coordinates at 0.5R, 1R, 2R, and 4R along the actual rotor thrust axis, including the surface normal, per-rotor far-wake pressure, expected wake speed, and JIRS-derived cushion force. This gives future live L2 validation a spatial pressure/velocity sampling contract instead of asking it to infer local wake quality from only whole-airframe totals.

`Aerodynamics4McL2PoweredHoverSurfaceWakeTransit` adds the corresponding transient arrival-time audit for those rotor-resolved surface probes. It brackets each probe's wake arrival between disk-plane induced velocity and far-wake velocity, records the configured dynamic-inflow time constant ratio, and derives a minimum transient sample-rate target for 0.5R, 1R, 2R, and 4R ground/ceiling clearances. The 5-inch curve reaches 98.2% Reynolds-supported thrust weight at its highest RPM but only 51.4% full support, while its lowest points remain below Re 10k. The same exporter compares axial-only BEMT with `COUPLED_LIFT_TORQUE_ANGULAR_MOMENTUM`, using the axial/tangential induction and Prandtl momentum relations documented in the [NREL AeroDyn theory manual](https://www.nrel.gov/wind/nwtc/assets/pdfs/ad-theory.pdf). Missing coherent wake rotation therefore does not explain the measured power deficit, which remains evidence for incomplete profile-drag, three-dimensional rotation, hub, surface-condition, or other viscous-loss modeling.

`RotorHoverPowerLossDecomposition` conditions the BEMT lift-induced power factor on measured thrust before subtracting modeled profile power from measured shaft power. With the low-Re extension, the remaining unresolved fraction spans 3.0-19.4%, and the conditional profile-power scale needed to absorb every non-lift loss spans 1.07-2.16 with no common monotonic curve across the two propellers. It is therefore diagnostic rather than a Cd correction; the separate Reynolds, three-dimensional, tip, and Mach corrections used in published low-Re electric-propulsion BEM work ([McCrink and Gregory](https://doi.org/10.2514/1.C033622)) should be investigated independently before any generalized loss model is fitted.

`SnelMcCrinkRotationalAugmentation` implements the optional normal-force correction from equation 9 of the [McCrink and Gregory low-Re electric-propeller paper](https://doi.org/10.2514/6.2015-3296), with the full derivation also available in the [public dissertation](https://etd.ohiolink.edu/acprod/odb_etd/etd/r/1501/10?clear=10&p10_accession_num=osu1449142886). It applies `1.5 (c/r)^2 (omega r / V_local)^2 (CL_potential - CL_2D)` to chord-normal force through `0.85R`, preserves the 2-D chordwise coefficient, and rotates the corrected coefficients back to CL/CD. The policy is disabled by default. With the low-Re polar, the 64-row sensitivity output raises CT by 0.00264-0.00897 and CP by 0.00151-0.00637, but corrected CP still remains 13.6-34.5% below measurement. At the highest 9x6.75 axial-only point, CP error improves from -19.5% to -17.6% while CT moves from -0.09% to +2.02%. The exporter retains corrected/source-limited/clamped counts and now adds thrust-weighted coverage, so this remains a partially explanatory physical mechanism rather than a fitted or accepted DA4002 correction.
